Description

In this lab I practiced doing analog output using PWM and sound output using a piezo buzzer. First I wired an RGB LED and used PWM to mix different colors. Then I used a joystick as analog input to dynamically change the RGB LED color. After that I wired a passive buzzer and used tone() to play individual notes, then expanded that into two melodies. Finally, in the Extra Volt section I combined both light and sound into a simple audio-visualizer where specific notes trigger specific RGB colors.

What it Does and How to Use It

  • Part 1: RGB LED displays different colors by changing PWM values on R/G/B pins.
  • Part 2: Joystick controls the RGB LED color directionally in real time.
  • Part 3: Passive buzzer plays a single note using tone().
  • Part 4: Passive buzzer plays a melody.
  • Extra Volt: RGB and buzzer make an audio-visualizer.

How it Works

The Arduino Uno does not output true analog voltage on its pins. Instead it uses Pulse Width Modulation on spesific pins. PWM rapidly switches a pin between HIGH and LOW. By changing thow long it stays HIGH, the average power delivered changes. LEDs respond to this as brightness, so analogWrite(0-255) makes the LED look dim to bright. An RGB LED is three LEDs inside one package that share a common ground. By changing the brightness of each color channel with PWM, the colors mix into many combinations like purple wich is red and blue. For sound, tone(pin, frequency) rapidly toggles a pin at a specific frequency, which gets the buzzer to create a pitch. A melody is made by playing multiple tones for different durations.

Mathematical Calculations

PWM mapping (joystick to LED brightness)

analogRead() returns 0-1023 and analogWrite() uses 0-255. I mapped the joystick readings into PWM values like this: 

pwm = map(raw, 0, 1023, 0, 255);

Frequency (tone) concept

tone(pin, frequency) uses the frequency in Hertz (Hz). Higher Hz = higher pitch. For example 260 Hz is a low-ish note. In Part 4, a melody is created by changing the frequency over time and holding each frequency for a duration (ms).

Part 2 — Dynamic Color Changer with Joystick

Procedure

  2. Connected joystick 5V to 5V and GND to GND.
  3. Connected joystick VRx to A0 and VRy to A1.
  4. Read joystick values with analogRead() and mapped them to PWM (0-255).
  5. Used joystick movement to control the RGB LED color channels.
  6. Recorded a demo video showing the color changing as the joystick moves.

const int X_pin = 0;  // VRx connected to A0
const int Y_pin = 1;  // VRy connected to A1

const int R_LED_PIN = 11;
const int G_LED_PIN = 10;
const int B_LED_PIN = 9;


void setup() {
  pinMode(R_LED_PIN, OUTPUT);
  pinMode(G_LED_PIN, OUTPUT);
  pinMode(B_LED_PIN, OUTPUT);
}


void loop() {
  int x = 0;
  int y = 0;
  x = analogRead(X_pin);              //Read VRx normally
  y = abs(1023-(analogRead(Y_pin)));  //Invert VRy read, as the module is inverted
  
  int redValue = 0;
  int greenValue = 0;
  int blueValue = 0;
  int xValue = 0;     //xValue is later reffered to as "douse," as it's main role is to douse the red and and blue LEDs

  xValue = (map(y, 512, 1023, 0, 255));             //Douse is the upper half of the Y axis
  xValue = constrain (xValue, 0, 255);              //Set the upper half range to 0 thru 255
  greenValue = (map(y, 0, 512, 0, 255));            //Set the green control to the lower half of the Y axis
  redValue = (map(x, 0, 512, 0, 255))-xValue;       //Set the red control to the left half of the X axis, then subtract the "Douse" value
  blueValue = (map(x, 512, 1023, 255, 0))-xValue;   //Set the blue control to the right half of the X axis, then subtract the "Douse" value

  blueValue = constrain(blueValue, 0, 255);         //Make sure the range of power for each output is 0 to 255
  redValue = constrain(redValue, 0, 255);
  greenValue = constrain(greenValue, 0, 255);

  analogWrite(R_LED_PIN, redValue);                       //Make the lights turn on.
  analogWrite(G_LED_PIN, blueValue);                     
  analogWrite(B_LED_PIN, greenValue);

  Serial.print("X-axis: ");                         //I'm letting this still execute, but nothing will display
  Serial.println((x));                              //until the Serial.begin(baud) is turned back on. 
  Serial.print("Y-axis: ");
  Serial.println((y));
  Serial.print("Red=");
  Serial.println(redValue);
  Serial.print("Blue=");
  Serial.println(blueValue);
  Serial.print("Green=");
  Serial.println(greenValue);
  Serial.print("Douse=");                           //Remember, "DOUSE" is the xValue and it's role is to turn off red and blue LEDs
  Serial.println(xValue);
  Serial.print("\n");

//  delay(500);                                     //Used in debugging so one can read the serial output. Keep off to keep LED control real-time
}

Problems Faced and Notes

  • The joystick Y-axis on my module was inverted, so pushing up decreased the value instead of increasing it. I corrected this by using: 
    y = abs(1023 - analogRead(Y_pin));
    This flipped the range so the physical motion matched the logical direction in code.
  • Getting smooth color blending was tricky because I divided the joystick axes into halves (0-512 and 512-1023). If I didn't carefully map and constrain each section, negative values would occur after subtracting the xValue. This caused flickering or unexpected color jumps.
  • The douse variable (xValue) was designed to reduce red and blue intensity when the joystick moves into the upper Y range. Without constraining the values after subtraction, the LED could receive invalid brightness levels. Using constrain() fixed this.
  • At first, colors looked incorrect because I accidentally swapped the analogWrite outputs. I verified which PWM pin controlled which RGB leg and corrected the mapping: 
    
    analogWrite(R_LED_PIN, redValue);
    analogWrite(G_LED_PIN, blueValue);
    analogWrite(B_LED_PIN, greenValue);
    Matching physical wiring to variable names was important.
  • The joystick center position does not sit exactly at 512. Small variations caused slight color shifts even when the joystick was centered. Adding constrain() and carefully mapping ranges reduced noticeable jitter.
  • Serial output was useful for debugging, but because Serial.begin() is currently commented out, nothing prints unless it is re-enabled. During debugging, I temporarily added a delay() so I could clearly read the values before removing it for real-time LED control.

#include "pitches.h"

const int full_note = 2000; // Full note duration in ms 

const int half_note = full_note / 2;
const int quarter_note = full_note / 4;
const int eighth_note = full_note / 8;
const int sixteenth_note = full_note / 16;

const int dotted_half_note = (half_note * 3) / 2;
const int dotted_quarter_note = (quarter_note * 3) / 2; 
const int dotted_eighth_note = (eighth_note * 3) / 2; 
const int dotted_sixteenth_note = (sixteenth_note * 3) / 2; 

const int BUZZER_PIN = 13;

void setup() {
  pinMode(BUZZER_PIN, OUTPUT); 
}

void loop() { 

  // Twinkle Twinkle Little Star
  // C C G G A A G
  tone(BUZZER_PIN, NOTE_C4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_C4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_G4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_G4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_A4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_A4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_G4, half_note);
  delay(half_note * 1.2);

  // F F E E D D C
  tone(BUZZER_PIN, NOTE_F4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_F4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_E4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_E4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_D4, quarter_note);
  delay(quarter_note * 1.2);

  tone(BUZZER_PIN, NOTE_D4, quarter_note);
  delay(quarter_note * 1.2);
  
  tone(BUZZER_PIN, NOTE_C4, half_note);
  delay(half_note * 1.2);

  noTone(BUZZER_PIN);
  delay(1200);
}

Problems Faced and Notes

  • At first the notes ran together and sounded messy, so I used a small pause multiplier (1.2x) after each note so you can hear separation.
  • The dotted note durations were useful, but using decimal math can be weird in Arduino, so for my second version I used integer math (*3 / 2) for dotted notes.
  • Using pitches.h made it way easier because I could use NOTE_C4, NOTE_G4, etc. instead of guessing frequencies.
  • Sometimes the buzzer sounded quiet until I re-seated the jumper wires and made sure GND was solid.
  • Keeping note durations as constants made it easy to adjust the tempo just by changing full_note.

How It Works

Each time a note plays using tone(), I set the RGB LED to a specific color using analogWrite(). This creates a simple audio-visualizer effect where pitch corresponds to color. Longer notes stay illuminated longer because the delay() is longer. Since PWM is still being used for brightness control, the LED changes instantly when a new note plays.

Problems Faced and Notes

  • The LED sometimes stayed the previous color if I forgot to explicitly set all three RGB channels before each note.
  • Keeping the structure identical to Part 4 made debugging much easier.
  • Assigning a unique color per note kept the logic simple and avoided overcomplicating the code.
