Description

In this lab I explored analog input on the Arduino using a potentiometer and a joystick. I measured a changing analog voltage with analogRead(), then mapped that value to LED output. Next, I read the joystick's X and Y axes and used thresholding to detect directional pushes. Finally, I built a Whack-A-Mole style game where one LED is the target and the joystick direction is used to hit the correct LED and a time is given at the end.

What it Does and How to Use It

  • Part 1: Turn the potentiometer knob and watch the raw ADC value + voltage print in Serial Monitor.
  • Part 2: Turn the potentiometer knob to control how many lined up LEDs turn on.
  • Part 3: Move the joystick and watch the X/Y values change in Serial Monitor. Press the joystick to change SW.
  • Part 4: Move the joystick up, down, left, right, and diagonals to light the matching LEDs in a plus layout.
  • Part 5: When the target LED lights up, move the joystick in the matching direction to hit it. A time is shown at the end of the game.

How it Works

The potentiometer and joystick axes are read through the Arduino's ADC (0-1023). The potentiometer is a voltage divider, the wiper outputs a voltage between 0V and 5V depending on its position. The joystick VRx and VRy behave the same way but are mechanically controlled by the stick position. The LED behaviors are driven with digital output pins controlling voltage. For the bar graph, the ADC reading is mapped to an integer count of LEDs to illuminate. For joystick direction, thresholds and a deadzone are used to convert analog values into discrete directions and in Part 4, diagonal directions light two LEDs at once.

Calculations

Potentiometer voltage conversion

The Arduino ADC returns 0-1023 for 0-5V. I converted readings to voltage using: 

voltage = raw * 5.0 / 1023.0

Mapping potentiometer range to LED count

To display a 5-LED bar, I mapped the potentiometer range to the number of LEDs to turn on: 

ledOn = map(potValue, 0, 1023, 0, 5)

Example: If potValue is about 512, ledOn is about 2-3 LEDs depending on rounding.

Part 2 — Map Potentiometer Value to LED Output

Procedure

  1. Wired 5 LEDs each with a 220Ω resistor to digital pins 2-6.
  2. Read the potentiometer value (0-1023) from A0.
  3. Used map() to convert the potentiometer value into how many LEDs should be on (0-5).
  4. Turned on that many LEDs from the array so the “bar” grows/shrinks when the knob turns.

Circuit / Schematic

Demo

Code

// LED pins (left to right)
int pin_LEDarray[] = {2, 3, 4, 5, 6};

// potentiometer analog pin
#define pin_potentiometer A0

void setup() {
  // set each LED pin as an output
  for (int i = 0; i < 5; i++) {
    pinMode(pin_LEDarray[i], OUTPUT);
  }
}

void loop() {
  // start by turning all LEDs off so we can redraw the bar cleanly
  for (int i = 0; i < 5; i++) {
    digitalWrite(pin_LEDarray[i], LOW);
  }

  // read potentiometer (0 to 1023)
  int potValue = analogRead(pin_potentiometer);

  // convert pot range to number of LEDs on (0 to 5)
  int ledOn = map(potValue, 0, 1023, 0, 5);

  // turn on the first ledOn LEDs
  for (int i = 0; i < ledOn; i++) {
    digitalWrite(pin_LEDarray[i], HIGH);
  }
}

Part 4 — Plus Layout LEDs Controlled by Joystick Direction

Circuit and Layout Photo

Circuit and Schematic

Demo

Code

int LEDarray[] = {2, 3, 4, 5, 6};

#define pin_JoyX A0
#define pin_JoyY A1

// thresholds
const int highThresh = 900;
const int lowThresh  = 100;
const int midLow     = 350;
const int midHigh    = 650;

void setup() {
  for (int i = 0; i < 5; i++) {
    pinMode(LEDarray[i], OUTPUT);
  }
}

void loop() {
  // turn off everything first
  for (int i = 0; i < 5; i++) {
    digitalWrite(LEDarray[i], LOW);
  }

  // read joystick
  int joyXValue = analogRead(pin_JoyX);
  int joyYValue = analogRead(pin_JoyY);

  // deadzone checks
  bool xMid = (joyXValue > midLow && joyXValue < midHigh);
  bool yMid = (joyYValue > midLow && joyYValue < midHigh);

  // directions
  bool up    = (joyXValue < lowThresh);
  bool down  = (joyXValue > highThresh);
  bool left  = (joyYValue < lowThresh);
  bool right = (joyYValue > highThresh);

  // center
  if (xMid && yMid) {
    digitalWrite(LEDarray[2], HIGH);
  }

  // diagonals
  else if (up && right) {
    digitalWrite(LEDarray[0], HIGH);
    digitalWrite(LEDarray[1], HIGH);
  }
  else if (up && left) {
    digitalWrite(LEDarray[0], HIGH);
    digitalWrite(LEDarray[3], HIGH);
  }
  else if (down && right) {
    digitalWrite(LEDarray[4], HIGH);
    digitalWrite(LEDarray[1], HIGH);
  }
  else if (down && left) {
    digitalWrite(LEDarray[4], HIGH);
    digitalWrite(LEDarray[3], HIGH);
  }

  // single directions (other axis centered)
  else if (up && yMid) {
    digitalWrite(LEDarray[0], HIGH);
  }
  else if (right && xMid) {
    digitalWrite(LEDarray[1], HIGH);
  }
  else if (down && yMid) {
    digitalWrite(LEDarray[4], HIGH);
  }
  else if (left && xMid) {
    digitalWrite(LEDarray[3], HIGH);
  }

  delay(10);
}

Explanation

I figured out why my first version felt wrong. I assumed the joystick VRx and VRy matched the same physical directions that my code was checking, but on my joystick module the axes were effectively rotated (up and down changed the X reading and left and right changed the Y reading). That meant my direction checks were correct in logic but wrong for my actual hardware orientation. Another issue was that diagonals require two directions to be true at the same time, so using a single else-if chain can make it impossible to light two LEDs unless the code is written to explicitly handle diagonal cases first. Once I mapped the joystick readings to the real directions and prioritized diagonals, the plus layout behaved consistently.

Debugging Notes

  • The joystick would sometimes flicker between directions near the center, so I used a deadzone range and midLow/midHigh checks.
  • My first thresholds were too close to the center, so diagonals were triggering randomly. I fixed it by printing raw X/Y values and increasing the high and low thresholds.
  • I had one LED not turning on because it was backwards in the breadboard. Flipping the LED polarity fixed it.
  • One direction LED was inconsistent because I accidentally used the wrong digital pin in the array compared to my wiring. I checked the pin numbers on the Arduino header and matched them to the LEDarray order.

Design Process

I started by defining what counts as each joystick direction using thresholds and a deadzone so the center doesn't accidentally trigger. Then I made the game loop run 20 rounds, each round picks a random direction LED, turns it on, and waits in a while-loop until the joystick matches that direction. When the input matches, the LED turns off and the next round starts. I timed the whole game using millis() by saving a start time before the rounds and subtracting it from the end time after 20 hits.

Code

const byte LEDarray[] = {2, 6, 5, 3};  
int middle = 4;

#define pin_JoyX A0
#define pin_JoyY A1

// joystick threshold
const int highThresh = 900;
const int lowThresh  = 100;
const int midLow     = 350;
const int midHigh    = 650;

unsigned long startTime;
unsigned long endTime;
unsigned long Time;

void setup() {
  // 4 direction LEDs as outputs
  for (int i = 0; i < 4; i++) {
    pinMode(LEDarray[i], OUTPUT);
  }

  // center LED output
  pinMode(middle, OUTPUT);

  Serial.begin(9600);
}

void loop() {
  Serial.println("Get ready ... :");

  // blink the center LED 2 times for .5 seconds
  digitalWrite(middle, HIGH);
  delay(500);
  digitalWrite(middle, LOW);
  delay(500);
  digitalWrite(middle, HIGH);
  delay(500);
  digitalWrite(middle, LOW);
  delay(500);

  // wait 1 second before starting
  delay(1000);

  // start the timer
  startTime = millis();

  int ledChoice;
  bool stickChoice = false;

  // run the game 20 times
  for (int roundNum = 0; roundNum < 20; roundNum++) {
    stickChoice = false;

    // pick a random LED number
    ledChoice = random(1, 5);

    // turn that LED on
    digitalWrite(LEDarray[ledChoice - 1], HIGH);

    Serial.print("LED target (1-4) = ");
    Serial.println(ledChoice);

    // wait until the joystick values match target direction
    while (stickChoice == false) {
      int joyXValue = analogRead(pin_JoyX);
      int joyYValue = analogRead(pin_JoyY);

      // 1 up (Y high)
      // 2 right (X high)
      // 3 down(Y low)
      // 4 left (X low)

      if (ledChoice == 1 && joyYValue > highThresh && (joyXValue > midLow && joyXValue < midHigh)) {
        stickChoice = true;
      }
      else if (ledChoice == 2 && joyXValue > highThresh && (joyYValue > midLow && joyYValue < midHigh)) {
        stickChoice = true;
      }
      else if (ledChoice == 3 && joyYValue < lowThresh && (joyXValue > midLow && joyXValue < midHigh)) {
        stickChoice = true;
      }
      else if (ledChoice == 4 && joyXValue < lowThresh && (joyYValue > midLow && joyYValue < midHigh)) {
        stickChoice = true;
      }
      else {
        stickChoice = false;
      }

      delay(10);
    }

    // once direction is hit turn off LEDs
    for (int j = 0; j < 4; j++) {
      digitalWrite(LEDarray[j], LOW);
    }

    // delay so it does not jump to next round
    delay(150);
  }

  // stop timer and print
  endTime = millis();
  Time = endTime - startTime;

  Serial.print("Took you ");
  Serial.println(Time);

  delay(3000);
}

Debugging Notes

  • The game would sometimes never detect a hit because my joystick was slightly off-center. I fixed it by keeping the midLow/midHigh deadzone check so it only counts a direction when the other axis is centered.
  • At first the LEDs would stay on after a hit because I only turned off the target LED. I changed it to turn off all four direction LEDs after each round so the next round starts clean.
  • The timing felt incorrect until I moved startTime = millis() to right before the rounds begin and endTime = millis() right after the loop ends.
  • Sometimes it would feel like the next round started too fast, so I added a small delay after turning off the LEDs so the player can see the hit.
  • When I tested, the center LED blink helped a lot because it gave a clear countdown and confirmed the board was running before the timer started.

Problems and Debugging Notes

  • Joystick readings were noisy near center, so I used a deadzone range.
  • Some LEDs were inconsistent until I re-checked the ground connections and digital pins.
  • Direction detection was fixed by printing raw X/Y values and adjusting thresholds.
  • For the plus layout, I checked LED polarity and confirmed each LED pin matched the LEDarray order.
  • For the game, I made sure the joystick direction checks required the other axis to be in the deadzone so wrong directions would not count.
